# DeepSeek 文件夹上传助手

一个用于 DeepSeek 网页端的批量上传插件，支持选择文件夹、批次注入附件、自动发送、手动发送、跳过不支持文件，并在 DeepSeek 输入区附近提供稳定的上传入口与会话内批次管理能力。

## 适用页面

- `https://chat.deepseek.com/*`
- `https://www.deepseek.com/chat*`
- `https://deepseek.com/chat*`

## 主要功能

- 支持选择整个文件夹或单独选择文件
- 自动过滤不支持的文件
- 支持大批量文件按批次上传
- 支持 `自动发送` 和 `仅注入` 两种模式
- 支持自定义扩展名
- 支持显示当前批次、待处理队列、已跳过文件、本次会话记录
- 支持批次间动态冷却、`服务器繁忙` 长退避和会话内自适应缩批
- 在能识别失败文件名时，支持 `仅重试失败项`
- 支持在 DeepSeek 原生上传按钮附近显示插件入口

## 安装方式

1. 打开 Chrome 扩展管理页：`chrome://extensions/`
2. 打开右上角“开发者模式”
3. 点击“加载已解压的扩展程序”
4. 选择当前插件目录
5. 加载完成后，刷新 DeepSeek 页面

如果更新了 `content.js` 或 `manifest.json`，需要先在扩展管理页点击“重新加载”，再刷新 DeepSeek 页面。

## 使用流程

1. 打开 DeepSeek 对话页
2. 找到插件按钮
3. 点击按钮后，按当前设置选择文件夹或文件
4. 插件会先过滤文件，再打开管理面板
5. 根据当前模式执行上传

### 按钮位置

- 正常情况下，插件按钮会贴近 DeepSeek 原生上传附件按钮
- 如果当前页面结构暂时没有识别成功，按钮会回退到右下角悬浮显示

## 两种上传模式

### 1. 自动发送

- 插件会按批次把文件注入到 DeepSeek 输入区
- 自动补充一段批次说明文字
- 自动尝试点击发送按钮
- 当前批次完成后，自动进入下一批

适合：

- 多批次资料连续上传
- 希望尽量减少人工操作

### 2. 仅注入

- 插件只负责把当前批次放进输入框
- 不自动点击发送
- 由你手动点击 DeepSeek 发送按钮
- 发送后插件会尝试自动识别“已完成”

适合：

- 你想先修改提示词再发送
- 你希望自己控制每一批发送时机

## 面板说明

### 队列

显示三部分内容：

- `当前批次`
  - 表示文件已经进入 DeepSeek 输入区
  - 当前状态可能是注入中、待发送、发送中、冷却中、暂停或需重试
- `待处理队列`
  - 表示还没有进入输入区的文件
- `本次会话记录`
  - 显示最近几批的状态、文件数、开始/发送/完成时间
  - 当某一批失败时，会记录失败类型和当前提示文案

### 已跳过

显示被过滤掉的文件及原因，例如：

- 超过 `100MB`
- 文件类型不支持

每个文件都可以点击“强制加入”重新放回队列。

### 设置

可以调整以下内容：

- 上传来源：文件夹模式 / 文件模式
- 上传方式：自动发送 / 仅注入
- 批大小
- 发送前延迟
- 基础批次冷却
- 全局前缀 Prompt
- 自定义扩展名

设置会保存在浏览器本地存储中。

补充说明：

- 设置里的 `批大小` 是你的目标批大小
- 如果连续命中 `服务器繁忙`，插件会在当前会话里临时减半批大小
- 连续成功若干批后，会逐步恢复到你设置的目标批大小
- `基础批次冷却` 只定义底座，实际等待时间还会叠加“上一批文件数 x 0.1 秒”

## 底部按钮说明

### 暂停

- 暂停自动推进
- 已经进入输入框的当前批次会保留

### 清空

- 清空插件内部队列
- 已经注入到 DeepSeek 输入框里的文件不会被页面自动移除

### 重试本批

- 重新执行当前批次的注入或发送流程

### 仅重试失败项

- 只在插件能可靠识别失败文件名时显示
- 会把失败文件前置成新的重试批次
- 同批其余未发送文件会回到后续待处理队列

### 继续

- 如果当前批次处于待发送状态，会显示为“标记已发”
- 如果当前批次已处理完成，则进入下一批
- 如果没有剩余任务，则关闭流程

## 文件过滤规则

- 单个文件超过 `100MB` 会被跳过
- 常见文档、文本、代码、表格、图片默认支持
- 可在设置中补充额外扩展名

## 自动填充的批次说明

在 `自动发送` 模式下，插件会自动写入一段批次说明，用来告诉 DeepSeek：

- 当前是第几批
- 这些文件属于同一个任务
- 是否还有后续批次
- 是否已经是最后一批

如果设置了“全局前缀 Prompt”，插件会把你的提示放在批次说明前面。

## 自动保护机制

### 1. 动态冷却

- 每批完成后不会立刻注入下一批
- 默认基础冷却为 `5 秒`
- 额外叠加 `上一批文件数 x 100ms`
- 例如上一批 `50` 个文件，则下一批默认约在 `10 秒` 后继续

### 2. `服务器繁忙` 长退避

- 普通附件确认重试会使用较短退避
- 如果识别到 `服务器繁忙 / server busy`，会切换到更长的退避节奏：`3s / 6s / 12s`

### 3. 会话内自适应缩批

- 连续命中 `服务器繁忙` `2` 次后，当前会话的有效批大小会减半
- 最低不会降到 `5` 个以下
- 连续成功 `2` 批后，会逐步恢复到你设置的目标批大小

### 4. 会话日志

- 每一批都会记录开始、发送、完成或失败状态
- 可以用来快速确认“哪一批真的发出去了”

## 状态说明

插件按钮会根据当前状态显示不同提示：

- `注入中`
- `待发送`
- `发送中`
- `冷却中`
- `已暂停`
- `需重试`
- `已完成`

右上角数字会显示：

- 当前批次进度
- 或剩余待处理数量

会话记录里还可能看到：

- `已发送`
- `人工确认`
- `失败`

## 推荐使用方式

### 场景 1：一次性少量文件

- 直接用 `自动发送`
- 基本不需要额外操作

### 场景 2：多批次资料分析

- 建议填写“全局前缀 Prompt”
- 推荐使用 `自动发送`
- 这样 DeepSeek 更容易把多批文件视为同一任务

### 场景 3：每批都要人工确认

- 使用 `仅注入`
- 每次手动调整提示词后再发送

## 常见问题

### 1. 文件已经进输入框了，为什么还显示在面板里？

因为它属于“当前批次”，不是“待处理队列”。

含义是：

- 文件已经进入输入区
- 但插件还在等待发送完成或确认完成

### 2. 发送后文件名还没有消失怎么办？

先等插件自动识别。

如果页面没有识别到发送动作，可以点击：

- `继续`
- 或 `标记已发`

这样插件会手动收尾，并进入下一批。

### 3. 发送按钮识别失败怎么办？

插件已经做了页面适配，但 DeepSeek 页面结构可能变化。

如果自动发送失败：

- 先手动发送当前批次
- 再点击“继续/标记已发”

### 4. 为什么有时会看到“仅重试失败项”？

这说明插件已经在当前失败批次里可靠识别到了失败文件名。

此时你可以：

- 直接 `重试本批`
- 或使用 `仅重试失败项`，把失败文件先补传，其余未发送文件留到后续队列

### 5. 冷却时间为什么会变长？

因为当前实现不是固定等待。

实际等待时间会同时受两部分影响：

- 你在设置里给的基础冷却
- 上一批文件数量带来的附加等待

如果插件检测到 `服务器繁忙`，还可能触发：

- 更长退避
- 当前会话临时缩批

### 6. 按钮为什么有时会回到右下角？

这是兜底逻辑。

当插件暂时无法可靠识别 DeepSeek 原生上传按钮位置时，会回退到右下角悬浮显示，而不是消失。

### 7. 修改代码后为什么效果没变？

通常是因为扩展还没重新加载。

请按顺序执行：

1. 在 `chrome://extensions/` 里重新加载扩展
2. 刷新 DeepSeek 页面

## 当前实现特点

- 以 DeepSeek 当前页面结构为目标做了适配
- 使用状态驱动方式管理批次和队列
- 已经补充了会话日志、动态冷却、长退避和自适应缩批
- 尽量避免“文件已发送但队列未清”“找不到发送按钮”“附件已注入但动画还在转”这类问题

## 已知限制

- `仅重试失败项` 目前按文件名匹配；如果同一批里存在重名文件，可能会把重名项一起纳入失败重试
- 当前还不支持刷新页面后的断点续传；刷新后需要重新选择资料
- 真实运行稳定性仍然受 DeepSeek 页面结构和服务器状态影响
