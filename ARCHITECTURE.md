# DeepSeek 文件夹上传助手架构说明

## 1. 设计目标

这个插件的目标不是“往网页里塞文件”这么简单，而是把一个原本非常脆弱的网页自动化流程，整理成一个：

- 能稳定维持批次上下文
- 能显式表达状态
- 能识别失败态并停止误发
- 能继续演进和局部替换

的状态驱动系统。

因此，这套架构遵循四个原则：

1. **状态优先于步骤**
   先定义插件内部真相，再定义怎么推进流程。
2. **网页识别与业务编排分离**
   DeepSeek DOM 会变化，不能让它污染整个上传逻辑。
3. **UI 只投影状态，不自创真相**
   所有视觉反馈都来自共享状态，而不是另一套隐式逻辑。
4. **入口足够薄**
   `content.js` 只负责启动，真正的逻辑下沉到可维护模块里。

## 2. 当前分层

### `dfs-core.js`
职责：
- 共享常量
- 全局状态 `state`
- UI 句柄 `ui`
- DOM 缓存
- 持久化
- 通用 helper

它回答的问题是：
**插件内部有哪些稳定真相。**

### `dfs-ui.js`
职责：
- 队列页、已跳过页、设置页渲染
- 头部 chip、进度卡片、角标、底部按钮状态
- 抽屉开关与状态可视化

它回答的问题是：
**当前状态要怎么被用户理解。**

### `dfs-dom-adapter.js`
职责：
- 输入框、上传控件、发送按钮、停止按钮识别
- 附件卡片识别与失败态识别
- 原生上传按钮贴靠
- composer 观察器和页面信号采集

它回答的问题是：
**如何从不稳定的 DeepSeek DOM 中读取可靠信号。**

### `dfs-workflow.js`
职责：
- 批次推进
- 自动发送 / 仅注入
- 附件确认与发送确认
- 暂停、继续、重试、清空
- 文件选择与拖拽导入

它回答的问题是：
**插件业务流程如何真正运转。**

### `dfs-shell.js`
职责：
- 样式注入
- 插件按钮与抽屉壳层创建
- 页面级事件 wiring
- UI 挂载

它回答的问题是：
**这个系统如何接到网页上。**

### `content.js`
职责：
- 极简入口
- 页面 ready 后调用 `mountUI`

它回答的问题是：
**什么时候启动。**

## 3. 数据流与控制流

当前主链路可以概括为：

`用户操作 -> workflow 修改 state -> ui 根据 state 重绘 -> dom-adapter 提供页面信号 -> workflow 再决定下一步`

也就是：

- `dfs-core.js` 保存真相
- `dfs-dom-adapter.js` 感知外部页面
- `dfs-workflow.js` 做业务决策
- `dfs-ui.js` 把状态翻译成界面
- `dfs-shell.js` 负责挂载与接线
- `content.js` 负责启动

这个结构的意义在于：
**状态、页面识别、流程推进、界面表达，各自只承担一类复杂性。**

## 4. 加载顺序

`manifest.json` 当前按以下顺序注入：

1. `dfs-core.js`
2. `dfs-ui.js`
3. `dfs-dom-adapter.js`
4. `dfs-workflow.js`
5. `dfs-shell.js`
6. `content.js`

这个顺序不是偶然的：

- 先建立共享运行时
- 再挂 UI 能力
- 再挂页面识别能力
- 再挂业务编排能力
- 最后挂壳层和入口

这样后面的层总能消费前面的共享能力。

## 5. 为什么要这样拆

### 如果是页面适配问题
优先看 `dfs-dom-adapter.js`

例如：
- 找不到发送按钮
- 停止生成识别失效
- 附件失败态没识别到

### 如果是业务流程问题
优先看 `dfs-workflow.js`

例如：
- 第二批抢跑
- 自动发送时机不对
- 暂停 / 恢复 / 重试语义不对

### 如果是显示问题
优先看 `dfs-ui.js`

例如：
- 抽屉文案和状态不一致
- 当前批次 / 待处理队列展示有误
- 角标、按钮、底部动作不合理

### 如果是共享状态或配置问题
优先看 `dfs-core.js`

## 6. 当前架构收益

- 单一大文件已经被拆成明确层次
- `content.js` 已退化为最薄入口
- 页面适配不再和业务流程混写
- UI 不再偷偷维护另一套逻辑
- 后续修 bug 时可以更快落到具体层级
- 以后加日志、诊断、测试，会有更自然的挂点

## 7. 当前边界

这套架构已经比早期版本健康很多，但它仍然是一个站点自动化插件，因此天然存在边界：

- DeepSeek DOM 改版时，`dfs-dom-adapter.js` 仍需适配
- 上传解析、服务器繁忙、会话节奏问题，仍受网页与后端影响
- 当前持久化更偏配置和历史，不是完整的任务恢复系统
- 还没有系统化自动测试

换句话说：
**架构已经工程化，但运行环境仍然是脆弱的第三方网页。**

## 8. 后续演进建议

当前不建议再继续盲目拆文件，下一阶段更有价值的是：

1. 给 `dfs-workflow.js` 增加最小调试日志
2. 给 `dfs-dom-adapter.js` 提供可读的诊断接口
3. 建立真实页面回归清单
4. 再考虑做最小测试或诊断面板

## 9. 维护规则

### 新增代码时
- 纯状态 / helper 放 `dfs-core.js`
- 网页识别放 `dfs-dom-adapter.js`
- 批次推进 / 等待 / 收尾放 `dfs-workflow.js`
- 视觉与渲染放 `dfs-ui.js`
- 壳层与挂载放 `dfs-shell.js`
- `content.js` 不再承载业务逻辑

### 修改代码时
- 先判断问题属于哪一层
- 不要为了快，把 DOM 猜测塞回 workflow
- 不要让 UI 自己维护真实状态
- 不要让入口重新变胖

## 10. 一句话总结

这套插件已经从“单文件脚本”演进为：

**一个以共享状态为核心、以 DOM 适配为边界、以 workflow 为业务中轴、以 UI 为状态投影、以 shell 为挂载层的网页自动化系统。**
