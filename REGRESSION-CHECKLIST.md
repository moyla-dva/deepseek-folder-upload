# DeepSeek 文件夹上传助手回归清单

这份清单面向真实页面手工回归，目标是确认当前版本的核心链路在 DeepSeek 页面上可用且行为一致。

## 使用前准备

1. 在 `chrome://extensions/` 里重新加载扩展
2. 刷新 DeepSeek 页面
3. 准备三组测试资料：
   - `小批测试`：2 到 5 个普通文本/文档文件
   - `大批测试`：30 到 60 个可上传文件
   - `失败测试`：至少一批里要能观察到 `服务器繁忙` 或明确的失败附件状态

## Case 1：基础入口与面板

步骤：

1. 打开 DeepSeek 对话页
2. 确认插件按钮是否贴近原生上传按钮
3. 点击插件按钮展开面板

预期：

- 能看到 `队列 / 已跳过 / 设置` 三个页签
- 如果贴靠失败，按钮会回退到右下角悬浮显示
- 面板可以正常打开和关闭

## Case 2：自动发送主链路

步骤：

1. 在设置里选择 `自动发送`
2. 选择 `小批测试`
3. 观察当前批次注入、等待发送、发送完成

预期：

- 文件进入 `当前批次`
- 附件确认完成后，状态进入 `待发送` 再自动转 `发送中`
- 发送完成后，当前批次从流程中收尾
- `本次会话记录` 至少新增 1 条，状态最终为 `已发送` 或 `人工确认`

## Case 3：仅注入主链路

步骤：

1. 在设置里选择 `仅注入`
2. 选择 `小批测试`
3. 等文件进入输入区后，手动点击 DeepSeek 发送

预期：

- 插件不会自动点击发送
- 手动发送后，插件会尝试自动识别完成
- 若页面未识别到完成，可以点击 `标记已发`
- 会话记录里应体现本批发送与完成

## Case 4：动态冷却与倒计时

步骤：

1. 在设置里保持 `自动发送`
2. 选择 `大批测试`
3. 观察第一批发送完成后，到第二批开始前的等待过程

预期：

- 会出现 `冷却中`
- 提示文案中的剩余秒数会动态倒计时
- 冷却结束后才开始下一批注入
- 上一批文件越多，冷却越长

## Case 5：会话内自适应缩批

步骤：

1. 使用 `大批测试`
2. 在真实环境中连续观察至少两次 `服务器繁忙`
3. 查看设置页和后续批次规模

预期：

- 连续两次 `服务器繁忙` 后，会话有效批大小减半
- 后续新批次会按更小批次推进
- 会话记录与错误文案里能看到缩批后的变化提示

## Case 6：仅重试失败项

步骤：

1. 在某一批里触发明确失败，且失败附件名可见
2. 进入错误态
3. 观察底部是否出现 `仅重试失败项`
4. 点击该按钮

预期：

- 只有在插件识别到可靠失败文件名时才显示该按钮
- 点击后，失败文件会成为新的当前重试批次
- 同批未失败文件会回到待处理队列
- 会话记录里原失败批次会保留为 `失败`

## Case 7：重试本批

步骤：

1. 在错误态点击 `重试本批`
2. 分别验证“输入区还有残留附件”和“输入区已清空”两种情况

预期：

- 如果输入区残留附件未清，插件会阻止直接重试并给出提示
- 如果输入区已清空，会重新注入当前批次
- 会话记录会继续更新，不会丢失之前的失败记录

## Case 8：暂停与恢复

步骤：

1. 在注入中或待发送状态点击 `暂停`
2. 再点击 `恢复流程`

预期：

- 注入中的批次会回退到待处理队列
- 待发送中的批次会保留在输入区
- 恢复后会继续当前正确状态
- 会话记录里应显示暂停和恢复后的状态变化

## Case 9：已知限制确认

步骤：

1. 准备同一批里重名的两个文件
2. 在可控范围内尝试触发失败恢复

预期：

- 当前版本对 `仅重试失败项` 仍按文件名匹配
- 若出现重名文件，需人工确认是否误并
- 该风险应视为已知限制，不算隐藏缺陷

## Case 10：文档一致性

步骤：

1. 对照 [README.md](/Users/vainve/obsidian/deepseek-folder-upload/README.md:1)
2. 检查实际页面行为是否与说明一致

预期：

- 面板说明、状态说明、底部按钮说明与真实行为一致
- 已知限制与当前实现一致

## 本轮建议通过标准

- `自动发送` 与 `仅注入` 主链路通过
- `冷却中` 倒计时可见且推进正常
- 会话记录能跟随批次状态变化
- `仅重试失败项` 在可识别场景下可用
- `服务器繁忙` 不会导致静默丢文件
